## ОБЗОР ПРОЕКТА

### Назначение
**One-Day-DnD** — это веб-приложение для проведения сессий Dungeons & Dragons в локальной сети (LAN) без подключения к интернету.

### Основные характеристики
- **Архитектура:** Client-Server с WebSockets
- **Платформа:** Node.js + React
- **БД:** SQLite3 (better-sqlite3)
- **Сетевой стек:** Express.js + Socket.io
- **Контрактное ограничение:** Один сервер = Одна party (группа)

### Точки входа
- **Для DM (Мастер):** `http://localhost:3000/dm`
- **Для Игроков:** QR-код или прямая ссылка из DM Dashboard

## УСТАНОВКА И НАСТРОЙКА

### Шаг 1: Клонирование/подготовка

```bash
# Перейти в корневую папку проекта
cd "/home/asm/Документы/One Day DnD/dnd-lan"
```

### Шаг 2: Установка зависимостей

```bash
# Установка корневых зависимостей
npm install

# Установка зависимостей сервера
npm --prefix server install

# Установка зависимостей клиента
npm --prefix client install
```

**Windows (PowerShell):**
```powershell
npm install
npm --prefix server install
npm --prefix client install
```

### Шаг 3: Инициализация БД

База данных инициализируется автоматически при первом запуске сервера:
- **Расположение:** `server/data/app.db` (или в `%LOCALAPPDATA%\dnd-lan` на Windows)
- **Схема:** `server/src/schema.sql`
- **Миграции:** Выполняются автоматически

### Шаг 4: Настройка окружения (опционально)

Создайте файл `.env` в корне проекта для переопределения переменных:

```bash
# Порт сервера (по умолчанию 3000)
PORT=3000

# Переменные для разработки
NODE_ENV=development

# Логирование
LOG_LEVEL=info

# Путь к данным БД
DND_LAN_DATA_DIR=/path/to/data

# CSRF токен (cookie имя)
CSRF_COOKIE=csrf_token

# CSP (Content Security Policy)
CSP_DISABLED=0
CSP_ALLOW_UNSAFE_EVAL=0

# CORS для разработки
DEV_ALLOWED_ORIGINS=http://localhost:5173

# Join код (опционально)
PREFLIGHT_JOIN_CODE=
```

## ЗАПУСК ПРИЛОЖЕНИЯ

### Режим разработки

```bash
npm run dev
```

**Что происходит:**
- Запускается сервер Express на `http://localhost:3000`
- Запускается Vite dev сервер на `http://localhost:5173`
- Hot Module Replacement (HMR) включен
- Доступ DM: `http://localhost:3000/dm` или `http://localhost:5173/dm`

### Режим производства

```bash
# 1. Собрать фронтенд
npm run build

# 2. Запустить сервер
npm run start
```

**Результат:**
- Клиент собран в `server/public`
- Сервер обслуживает статику на порту 3000
- Одна точка входа: `http://localhost:3000/dm` или `/player`

### Присоединение игроков

1. DM открывает Dashboard
2. Генерируется QR код или ссылка для присоединения
3. Игроки переходят по ссылке или сканируют QR
4. DM подтверждает присоединение игрока
5. Игрок получает доступ к интерфейсу

## БАЗА ДАННЫХ

### Схема БД (`server/src/schema.sql`)

```sql
-- Основные таблицы
- parties           -- Группы D&D
- players           -- Игроки в party
- characters        -- Персонажи игроков
- inventory         -- Инвентарь персонажей
- bestiary          -- Монстры и враги
- maps              -- Карты сессий
- backup_history    -- История резервных копий
- sessions          -- Сессии пользователей
- events            -- События party
```

### Важные ограничения

**Single-Party Contract:**
- Один сервер обслуживает только одну party
- При старте сервер проверяет количество party в БД
- Если `COUNT(parties) > 1` — сервер не стартует
- Импорт backup с несколькими party отклоняется

### Миграции

Миграции хранятся в `server/src/migrations/` и выполняются автоматически:
```javascript
runMigrations(db); // Вызывается при инициализации БД
```

### Резервные копии

- **Тип:** ZIP архивы
- **Расположение:** `server/data/backups/` или `%LOCALAPPDATA%\dnd-lan\backups\`
- **Автоматическое сохранение:** Каждые 30 минут (при активности)
- **API:** `GET /api/backup` — скачать, `POST /api/backup` — восстановить

## API МАРШРУТЫ

### Аутентификация

```
POST /api/auth/login
  Body: { username, password }
  Response: { token, dmId }

POST /api/auth/logout
  Response: { success: true }
```

### Party управление

```
GET /api/party
  Response: { id, name, players[] }

POST /api/party/create
  Body: { name }
  Response: { partyId }

PUT /api/party/:id
  Body: { name, description }
```

## ЗАМЕЧАНИЯ О СЕТЕВОЙ АРХИТЕКТУРЕ

### WiFi подключение для игроков

1. Сервер запущен на ноутбуке DM (порт 3000)
2. Все остальные подключаются в одной WiFi сети
3. Клиент открывает `http://<dm-ip>:3000/player`
4. WebSocket соединение в реальном времени

### Адреса доступа

```
DM на машине:    http://localhost:3000/dm
DM извне:        http://<machine-ip>:3000/dm
Игрок на машине: http://localhost:3000/player
Игрок извне:     http://<machine-ip>:3000/player
```

### Требования сети

- Отсутствие необходимости интернета
- Локальная LAN сеть
- Порт 3000 открыт между машинами
- Stable Wi-Fi соединение рекомендуется

## ВАЖНЫЕ ОГРАНИЧЕНИЯ И СОГЛАШЕНИЯ

### Single-Party Contract
```javascript
// Сервер обслуживает ровно одну party
// Проверка при старте:
SELECT COUNT(*) FROM parties;
if (count > 1) {
  throw new Error("multiple_parties_not_supported");
}
```

### Валидация данных
- Все входящие JSON проверяются
- Максимальный размер: 2 МБ
- UTF-8 кодировка обязательна

### Безопасность
- CSRF токены на всех POST/PUT/DELETE
- CORS только для localhost (dev)
- Rate limiting на некритичные routes
- Helmet.js для HTTP заголовков
- Пароли хешируются bcryptjs

### Логирование
- Все чувствительные данные redact'ятся
- Cookie, Authorization, X-Player-Token скрываются
- Используется Pino для логирования

```bash

```

### Добавление новой API маршрута

1. Создать файл в `server/src/routes/newroute.js`
2. Импортировать в `server/src/bootstrap/app.js`
3. Зарегистрировать с `app.use('/api/newroute', ...)`
4. Добавить клиентский API в `client/src/api/`
5. Написать тесты в `server/test/`
6. Добавить E2E если нужно

### Добавление новой миграции БД

1. Создать файл в `server/src/migrations/` с временной меткой
2. Добавить функцию `export function migrate(db) { ... }`
3. Миграция выполнится автоматически при старте сервера
